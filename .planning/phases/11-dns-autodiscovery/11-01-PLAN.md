### 2. Publish records at DNS provider

At whatever manages `skale.club` (Cloudflare / Registro.br / etc.) add:

| Type | Name | Value | TTL |
|---|---|---|---|
| A | `mail` | `<VPS_IP>` | 300 |
| MX | `@` | `10 mail.skale.club.` | 3600 |
| TXT | `@` | `"v=spf1 a mx ip4:<VPS_IP> ~all"` | 3600 |
| TXT | `skaleclub._domainkey` | `"v=DKIM1; k=rsa; p=<public_key>"` | 3600 |
| TXT | `_dmarc` | `"v=DMARC1; p=none; rua=mailto:dmarc@skale.club; fo=1"` | 3600 |
| CNAME | `autoconfig` | `mail.skale.club.` | 3600 |
| CNAME | `autodiscover` | `mail.skale.club.` | 3600 |

**DKIM note:** Cloudflare and some providers split long TXT records automatically at 255 chars. If manual, split the public key into chunks of ≤255 chars each.

### 3. Set reverse DNS (Hetzner)
In Hetzner Cloud Console → Server → Networking → Reverse DNS:
- IPv4: set PTR to `mail.skale.club.`
- IPv6 (if enabled): same

### 4. Wait and verify propagation
```bash
# Run each from external network (or use dig @1.1.1.1)
dig +short mail.skale.club A
dig +short skale.club MX
dig +short skale.club TXT | grep spf
dig +short skaleclub._domainkey.skale.club TXT
dig +short _dmarc.skale.club TXT
dig +short autoconfig.skale.club CNAME
dig -x <VPS_IP>   # reverse DNS
```

All must return the expected value. If any fails → wait longer (TTL) or fix at registrar.

### 5. Verify via in-app endpoint
```bash
# Get domain ID
curl -H "Authorization: Bearer <token>" https://mail.skale.club/api/domains

# Trigger verification
curl -X POST -H "Authorization: Bearer <token>" \
  https://mail.skale.club/api/domains/<id>/verify
```

Expected: `spf.found=true`, `dkim.found=true`, `dmarc.found=true`, `mx.found=true`.

### 6. Verify autoconfig endpoint
```bash
# From external
curl -i "https://autoconfig.skale.club/mail/config-v1.1.xml?emailaddress=user@skale.club"
# or
curl -i "https://mail.skale.club/.well-known/autoconfig/mail/config-v1.1.xml?emailaddress=user@skale.club"
```

Expected: `200 OK`, XML starting with `<clientConfig version="1.1">`, containing `<hostname>mail.skale.club</hostname>` and both IMAP `<port>993</port><socketType>SSL</socketType>` and SMTP `<port>587</port><socketType>STARTTLS</socketType>`.

### 7. End-to-end Thunderbird test
- Open Thunderbird (local, not VPN)
- Add account → enter `<any-user>@skale.club` + password
- Click Continue → "Looking up configuration: ISP..." → success
- Confirm servers shown: IMAP mail.skale.club:993 SSL, SMTP mail.skale.club:587 STARTTLS
- Click Done — Thunderbird connects (Phase 10 must be complete for TLS; otherwise expect cert warning)

## Verification checklist (DNS-01, DNS-02, DISCO-01)
- [ ] `dig` returns all 7 records correctly
- [ ] `POST /api/domains/:id/verify` → all `verified`
- [ ] Reverse DNS resolves to `mail.skale.club.`
- [ ] Autoconfig XML endpoint returns valid clientConfig
- [ ] Thunderbird autodiscovery finds settings without manual entry

## Risk
- DNS propagation delay — plan 1h buffer
- Wrong public key format — strip `-----BEGIN PUBLIC KEY-----` headers, use only base64 content
- Caddy needs `autoconfig.skale.club` block too (or wildcard) — update `/etc/caddy/Caddyfile`
