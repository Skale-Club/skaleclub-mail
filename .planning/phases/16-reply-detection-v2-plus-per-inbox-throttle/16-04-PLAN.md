<objective>
One small hardening to make the "midnight UTC reset" contract explicit:

**Explicit UTC timezone on the midnight cron** — `src/server/jobs/index.ts` currently has `cron.schedule('0 0 * * *', () => resetDailyLimits().catch(...))`. node-cron's third argument accepts `{ timezone: 'UTC' }`. Without it, the schedule runs at the machine's local-time midnight. In the alpine container TZ is unset → defaults to UTC, so today this is fine — but it's not GUARANTEED. Operators can set `TZ` env at any time (e.g. setting `TZ=America/Sao_Paulo` to make `cleanupOldMessages` run at local 3am would also shift the reset). Making the contract explicit prevents a future "why did the reset run at the wrong time" incident.

Purpose: One-arg defensive change. Costs effectively no execution time. Closes the only remaining hand-waving spot in the throttle pipeline (the daily-reset "guarantee" depends on `0 0 * * *` running at UTC midnight). Independent file → safe to land in Wave 1 parallel with Plans 16-01 and 16-03.
Output: One file, two single-line edits (cron options + startup log).
</objective>
