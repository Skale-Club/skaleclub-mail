<objective>
Wire the per-inbox throttle into the outreach processor loop. Three concrete behaviors land in `src/server/jobs/processOutreachSequences.ts`:

1. **Rate-limit filter at lead-selection time** — call the Plan 16-01-extended `canSendFromAccount(emailAccount, now)` and route the existing "skip" branch to emit a structured `rate_limit_per_inbox` log instead of the current generic "has reached daily limit" string (which misleadingly suggests the daily cap is the cause).
2. **Jitter the next pending lead's nextScheduledAt** — after a successful send, find the NEXT same-inbox pending lead (in this tick's already-loaded `pendingLeads` array — no extra query) and bump its `nextScheduledAt` via `applySendJitter(emailAccount.minMinutesBetweenEmails, emailAccount.maxMinutesBetweenEmails)`. This prevents the per-inbox burst pattern that trips Gmail/Yahoo rate-limit detection.
3. **Standardize skip reasons** — replace the existing ad-hoc `console.log`/`console.error` skip messages with the standardized shape `[outreach.processor] {action, campaignId?, leadId?, emailAccountId?, reason?}`. Reasons enumerated per CONTEXT.md: `rate_limit_per_inbox`, `daily_limit_reached`, `suppression`, `no_active_step`, `outside_send_window`, `claim_conflict`, `unsubscribed`, `campaign_inactive`, `no_account`, `account_not_verified`.

Purpose: This is the single biggest deliverability fix in the system today. Without it, the processor sends every eligible lead for an inbox in a single tick — Gmail / Yahoo / iCloud spam-detection sees the burst, classifies as bulk, downgrades the sender reputation. The throttle keeps each inbox at a human-paced send cadence (default 5 min between sends per the schema's minMinutesBetweenEmails default).
Output: One file edited. No new files. No schema changes (Plan 16-01 handled both). The advisory-lock wrapper from Phase 14 is preserved exactly as-is.
</objective>
