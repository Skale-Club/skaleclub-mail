<objective>
Add a module-level concurrency guard to the outreach sequence processor cron in src/server/jobs/index.ts. An overlapping invocation (cron fires while previous run is still in progress) must be skipped with a log message, not stacked.

Purpose: The sequence processor runs every 5 minutes. If a run takes more than 5 minutes (large campaign, slow SMTP), the next tick starts a second parallel run. Without a guard, multiple overlapping runs compete for the same DB rows, risk double-sending, and exhaust connection pools.
Output: jobs/index.ts with isSequenceProcessing flag; REQUIREMENTS.md with QUAL-04 marked complete.
</objective>

  <action>
    **Step A — Add module-level flag to jobs/index.ts:**

    After all the import statements (after line 7, before the `export function startJobs()` declaration), add one blank line followed by:

        let isSequenceProcessing = false

    This must be at module scope, NOT inside the startJobs function and NOT inside the cron.schedule callback. Placing it inside either would reset it on every call or every tick, defeating the guard.

    **Step B — Replace the processOutreachSequences cron handler:**

    Find the existing block (currently lines 28-30):

        // Process outreach sequences every 5 minutes
        cron.schedule('*/5 * * * *', () => {
            processOutreachSequences().catch((err) => console.error('[jobs] processOutreachSequences failed:', err))
        })

    Replace it with:

        // Process outreach sequences every 5 minutes
        cron.schedule('*/5 * * * *', () => {
            if (isSequenceProcessing) {
                console.log('[jobs] processOutreachSequences already running, skipping tick')
                return
            }
            isSequenceProcessing = true
            processOutreachSequences()
                .catch((err) => console.error('[jobs] processOutreachSequences failed:', err))
                .finally(() => { isSequenceProcessing = false })
        })

    Do NOT change any other cron.schedule blocks. Do NOT add guards to processQueue, processHeld, cleanupMessages, resetDailyLimits, processReplies, or processBounces — only processOutreachSequences gets the guard in this phase.

    **Why `.finally` is mandatory:** `.catch()` alone resets the flag only on error. If the promise resolves successfully, a `.catch()`-only pattern would leave the flag stuck at `true` permanently after the first successful run. `.finally()` runs unconditionally whether the promise resolves or rejects.
