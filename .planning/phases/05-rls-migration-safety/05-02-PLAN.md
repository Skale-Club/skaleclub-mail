# Phase 05 Plan 02: Safe Index Migration Workflow (DBS-02, DBS-03)

## Objective

Establish the infrastructure for safely creating indexes without blocking writes: a `sql/indexes.sql` file with `CREATE INDEX CONCURRENTLY` statements, an `npm run db:indexes` script to execute it, and an index health verification script.

**Purpose:** `CREATE INDEX` (non-concurrent) acquires a SHARE lock that blocks all writes for the duration. For production data, this is unacceptable. `CREATE INDEX CONCURRENTLY` allows writes during index creation but must NOT run inside a transaction. The existing `db:push` (Drizzle) wraps in transactions — separate SQL file + script is required.
**Output:** `sql/indexes.sql`, `scripts/verify-indexes.ts`, updated `package.json` scripts.

## Context

@package.json
@supabase/migrations/013_add_performance_indexes.sql
@src/db/index.ts

### Why CONCURRENTLY

- `CREATE INDEX` holds `SHARE LOCK` — blocks all INSERT/UPDATE/DELETE on the table
- `CREATE INDEX CONCURRENTLY` only holds weaker locks — allows concurrent writes
- CONCURRENTLY cannot run inside a transaction block — `db:push` wraps transactions
- Solution: separate SQL file executed via `psql` (no transaction wrapping)

### Existing State

- `013_add_performance_indexes.sql` uses `CREATE INDEX IF NOT EXISTS` (non-concurrent) — this is the OLD approach
- No `sql/` directory exists
- No `db:indexes` script in package.json
- `scripts/run-rls-migration.ts` exists as a precedent for running SQL files programmatically

Two changes:

**A. Update package.json** — add `db:indexes` script:
```json
"db:indexes": "psql \"$DATABASE_URL\" -f sql/indexes.sql"
```
This runs the SQL file directly via psql (outside any transaction wrapper), which is required for CONCURRENTLY. Add it after the existing `db:push` entry.

**B. Create `scripts/verify-indexes.ts`** — index health verification script that:
1. Connects to the database using `DATABASE_URL` from env
2. Queries `pg_index` joined with `pg_class` and `pg_namespace` to find all indexes in the `public` schema
3. Checks `indisvalid` for each index — if `false`, the index creation failed (e.g., CONCURRENTLY was interrupted)
4. For invalid indexes:
   - Log the index name and table
   - Drop the invalid index: `DROP INDEX CONCURRENTLY IF EXISTS {index_name}`
   - Re-create it from the `sql/indexes.sql` file (parse the CREATE INDEX statement for that index)
   - Re-check `indisvalid` after re-creation
5. Print summary: total indexes, valid count, invalid count (should be 0 after fix)
6. Exit with code 0 if all valid, 1 if any remain invalid after retry

The script should use `postgres` library (already in dependencies, used in `src/db/index.ts`) for the connection, not `psql` CLI.
