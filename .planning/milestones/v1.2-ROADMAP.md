# Roadmap: SkaleClub Mail — Thunderbird-Ready Production Mail (v1.2)

**Milestone:** v1.2 — Mail Server Production Readiness
**Created:** 2026-04-15
**Deploy target:** Hetzner VPS (Docker + Caddy + GitHub Actions)
**Goal:** An end-user can configure `user@skale.club` in Thunderbird / Outlook / Apple Mail and send/receive email reliably from/to the public internet.
**Phases:** 4 (10 through 13)

---

## Context

The full IMAP, SMTP submission, and MX receiver are already implemented in code (v1.1 mail server improvements — merged 2026-04-15, commit `8316a86`).

**What works today:**
- IMAP server (RFC 3501): TLS-capable, STARTTLS, SASL PLAIN/LOGIN, IDLE, APPEND, EXPUNGE, UID operations
- SMTP submission: TLS-capable, rate-limit, allocateNextUid atomic
- MX receiver (`mx-server.ts`) on port 25: accepts inbound, rejects non-local/non-routed recipients
- Autodiscovery endpoints: Thunderbird XML, Outlook XML, Apple `.mobileconfig`
- UI: dynamic connection-info card in SettingsPage
- Domain verification endpoint checks DNS records (SPF/DKIM/DMARC/MX) but does NOT sign/verify in-flight email
- Hetzner deploy via `.github/workflows/deploy-hetzner.yml`, Docker container exposing ports 9001/25/587/993, Caddy reverse-proxies only HTTP

**What's missing for Thunderbird to actually work in production:**
1. TLS certificates inside the Docker container for mail ports (Caddy only handles 443)
2. DNS records published correctly (SPF, DKIM, DMARC, MX, autoconfig CNAME)
3. DKIM signing applied to outbound mail (chave existe no DB mas não é usada em runtime)
4. SPF/DMARC/ARC verification on inbound MX
5. Hetzner port 25 unblocked (blocked by default for spam prevention)
6. MX hardening: rate-limit, greylisting, size/header validation

---

## Phases

- [ ] **Phase 10: TLS Certificates for Mail Ports** — Let's Encrypt certs mounted into container, renewal automation, env wiring
- [ ] **Phase 11: DNS + Autodiscovery** — publish MX/A/SPF/DKIM/DMARC + autoconfig CNAME records, verify via `/verify` endpoint
- [ ] **Phase 12: DKIM Signing + SPF/DMARC Verification** — wire `dkim_private_key` from DB into outbound signing; add `mailauth` inbound verification with Authentication-Results header
- [ ] **Phase 13: MX Hardening + Port 25 Unblock** — Hetzner support ticket, rate-limit/greylisting per connecting IP, max recipients, size enforcement, spamhaus DNSBL check

## Phase Details

### Phase 10: TLS Certificates for Mail Ports
**Goal:** Thunderbird connects to `mail.skale.club:993` via SSL/TLS and `mail.skale.club:587` via STARTTLS without cert warnings
**Depends on:** Phase 11 DNS records (A record for `mail.skale.club` must resolve to the VPS for Let's Encrypt HTTP-01)
**Blocks:** nothing — required for Thunderbird
**Success Criteria:**
1. `/etc/letsencrypt/live/mail.skale.club/fullchain.pem` and `privkey.pem` exist on the Hetzner host
2. Docker container mounts `/etc/letsencrypt:/etc/letsencrypt:ro` as read-only volume
3. `MAIL_TLS_CERT_PATH=/etc/letsencrypt/live/mail.skale.club/fullchain.pem` and `MAIL_TLS_KEY_PATH=...privkey.pem` are set as container env vars
4. Server logs on startup show "TLS certificate loaded" and IMAP listens in "SSL/TLS (implicit)" mode
5. `openssl s_client -connect mail.skale.club:993 -servername mail.skale.club` shows valid chain
6. Thunderbird connects without "untrusted certificate" prompt
7. Certbot systemd timer renews automatically; container reads new cert on reload

**Requirements:** TLS-01, TLS-02, TLS-03

### Phase 11: DNS + Autodiscovery
**Goal:** Thunderbird autoconfig succeeds from `user@skale.club` with zero manual server entry
**Depends on:** nothing (prereq for Phase 10 and 12)
**Blocks:** Phase 10 (Let's Encrypt HTTP-01 needs A record), Phase 12 (DKIM record publishing)
**Success Criteria:**
1. `dig mail.skale.club A` resolves to the Hetzner VPS IP
2. `dig skale.club MX` returns `10 mail.skale.club.`
3. `dig skale.club TXT` includes SPF record: `v=spf1 ip4:<vps-ip> ~all` (or `mx ~all`)
4. `dig skaleclub._domainkey.skale.club TXT` returns the DKIM public key from the `domains` table
5. `dig _dmarc.skale.club TXT` returns `v=DMARC1; p=quarantine; rua=mailto:dmarc@skale.club`
6. `dig autoconfig.skale.club CNAME` returns `mail.skale.club.`
7. `POST /api/domains/:id/verify` on the registered `skale.club` domain returns all statuses = `verified`
8. `GET http://autoconfig.skale.club/.well-known/autoconfig/mail/config-v1.1.xml?emailaddress=user@skale.club` returns valid XML served by the app

**Requirements:** DNS-01, DNS-02, DISCO-01

### Phase 12: DKIM Signing + SPF/DMARC Verification
**Goal:** Outbound mail signed with DKIM (Gmail/Outlook accept it); inbound mail authenticated (spoofers rejected or marked)
**Depends on:** Phase 11 (DNS records published)
**Blocks:** production usability (spam bucket without this)
**Success Criteria:**
1. `smtp-server.ts relayMessage()` fetches `dkim_private_key` + `dkim_selector` from the sender's domain in DB and passes to `nodemailer.createTransport({ dkim: {...} })`
2. Outbound email sent to `check-auth@verifier.port25.com` returns "pass" for SPF, DKIM, DMARC
3. Gmail test account receives email with "signed-by: skale.club" in message details
4. `mx-server.ts` integrates `mailauth` library: parses inbound, appends `Authentication-Results: mail.skale.club; spf=...; dkim=...; dmarc=...` header
5. Emails with `dmarc=fail` + DMARC policy `reject` are rejected at `onData` (550 response)
6. Emails with `dmarc=fail` + policy `quarantine` are delivered but flagged (`is_spam=true` or folder `spam`)
7. `outreach-sender.ts` also applies DKIM signing (same code path preferred)

**Requirements:** DKIM-01, AUTH-01, AUTH-02

### Phase 13: MX Hardening + Port 25 Unblock
**Goal:** Port 25 is reachable from the internet and the MX receiver survives the first 48h without being flooded by spam
**Depends on:** Phase 12 (so that rejected mail is clearly identified)
**Blocks:** nothing — this is the final production gate
**Success Criteria:**
1. Hetzner support ticket opened and port 25 unblocked (default block for new VPS); confirmed by `nc -vz mail.skale.club 25` from external network
2. `mx-server.ts` applies per-IP rate-limit at `onConnect` (max 10 connections/min per IP)
3. Greylisting implemented: first (IP, from, to) triple gets `451 4.7.1 Try again later`, accepted on retry ≥5min later
4. Size enforcement: reject > 25MB at `SMTPServer({ size })` option
5. Max recipients per session: 50 (smtp-server `maxRecipients`)
6. Spamhaus DNSBL check in `onConnect`: reject listed IPs with `554 5.7.1 Blacklisted`
7. After 48h in production: no more than 5% of legit mail rejected (false positive), no more than 100 spam messages/day in INBOX

**Requirements:** OPS-01, MX-01, MX-02

## Dependencies Graph

```
Phase 11 (DNS) ──┬──► Phase 10 (TLS via Let's Encrypt HTTP-01)
                 │
                 └──► Phase 12 (DKIM/SPF/DMARC needs records published)
                                    │
                                    └──► Phase 13 (MX hardening)
```

Phase 11 is the prerequisite — publish DNS records first, then certs + signing + hardening in parallel.

## Deferred / Out of Scope

- IDLE FLAGS/EXPUNGE push (EXISTS already works)
- LIST-EXTENDED, CONDSTORE, QRESYNC IMAP extensions
- POP3 server (IMAP sufficient for Thunderbird)
- CardDAV/CalDAV
- Sieve filtering (ManageSieve on port 4190)
- Pagination/streaming FETCH for huge mailboxes
- Raw RFC 2822 persistence (currently reconstructed)
- Webmail folder count hook invalidation after external IMAP mutations
- ESLint config (pre-existing)
- Supabase migration history drift (015–017 not pulled locally)

These remain in `.planning/BACKLOG-mail-server.md`.
